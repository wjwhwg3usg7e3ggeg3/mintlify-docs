Webhooks permitem que seu sistema receba notificações **em tempo real** sempre que um evento ocorrer.

Em vez de ficar consultando a API repetidamente para verificar mudanças, seu servidor recebe automaticamente um **HTTP POST** quando algo acontece.

Eventos comuns incluem:

- pagamento aprovado
- pagamento criado
- pagamento cancelado
- atualizações de status

---

# Como funciona

O fluxo de webhooks segue estas etapas:

1. um evento acontece na plataforma  
2. o sistema gera um payload com os dados do evento  
3. uma assinatura HMAC é criada para segurança  
4. um **HTTP POST** é enviado para sua URL configurada  
5. seu servidor recebe os dados e processa o evento  

---

# Configuração

Para receber webhooks você precisa configurar uma **URL pública** no seu sistema.

Exemplo:

```
https://seusite.com/webhook
```

Sempre que um evento ocorrer, a plataforma enviará uma requisição para essa URL.

---

# Método da Requisição

Todos os webhooks são enviados usando:

```
POST
```

---

# Headers enviados

Cada webhook inclui headers de segurança.

| Header | Descrição |
|------|-----------|
Content-Type | sempre `application/json` |
User-Agent | identificador da API |
X-Webhook-Timestamp | timestamp do envio |
X-Webhook-Signature | assinatura HMAC do payload |

Exemplo:

```
Content-Type: application/json
User-Agent: Vision-Wallet-API/1.0
X-Webhook-Timestamp: 1710000000
X-Webhook-Signature: sha256=9c0b9eab1f9a0d1...
```

---

# Estrutura do Payload

O corpo da requisição contém os dados do evento.

```json
{
  "type": "payment",
  "event": "payment.completed",
  "data": {
    "id": "pay_abc123",
    "value": 15000,
    "status": "COMPLETED"
  },
  "timestamp": 1710000000000
}
```

# Eventos disponíveis

| Evento | Descrição |
|------|-----------|
payment.created | pagamento criado |
payment.completed | pagamento confirmado |
payment.updated | atualização de status |

---

# Segurança

Todos os webhooks incluem uma **assinatura criptográfica** para garantir autenticidade.

A assinatura é gerada utilizando:

```
HMAC SHA256
```

Usando sua **API Key** como chave secreta.

---

# Exemplo de geração de assinatura

A assinatura é criada a partir do payload JSON.

Processo:

```
1 converter payload para JSON string
2 gerar HMAC SHA256 usando API Key
3 retornar assinatura em formato hex
```

Exemplo de resultado:

```
sha256=8c3fa21d7c1baf0a8f0c...
```

---

# Validação da assinatura

Seu servidor deve verificar se o webhook realmente veio da plataforma.

Processo:

1. receber payload
2. gerar assinatura local usando sua API Key
3. comparar com o header `X-Webhook-Signature`

Se forem iguais, o webhook é válido.

# Resposta esperada

Seu servidor deve responder com **status HTTP 2xx** para confirmar que recebeu o webhook.

Exemplo:

```http
HTTP/1.1 200 OK
```

Se seu servidor retornar erro, o webhook poderá ser reenviado.

---

# Sistema de retentativas

Se o webhook falhar, a plataforma tentará enviar novamente.

Estratégia:

- máximo de **5 tentativas**
- **backoff exponencial**

Exemplo de delays:

| Tentativa | Delay |
|------|------|
1 | 2 segundos |
2 | 4 segundos |
3 | 8 segundos |
4 | 16 segundos |
5 | 32 segundos |

Se todas falharem, o webhook será marcado como **failed**.

---

# Timeout

Cada webhook possui limite de resposta:

```
10 segundos
```

Se o servidor não responder dentro desse tempo, será considerado falha.

---

# Boas práticas

- sempre responder rapidamente ao webhook  
- validar a assinatura HMAC  
- processar eventos de forma idempotente  
- armazenar IDs de eventos para evitar duplicações  
- retornar status `200` após processar com sucesso  

---

# Fluxo recomendado

Fluxo ideal de processamento:

```
1 receber webhook
2 validar assinatura
3 verificar evento
4 processar dados
5 responder HTTP 200
```
